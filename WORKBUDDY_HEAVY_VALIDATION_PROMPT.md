# 给 WorkBuddy 的追加任务：历史事件重型验证与脏活清单

你正在实现《往哲荣耀 / Philosophy Auto Chess》的历史事件系统。继续以现有 `WORKBUDDY_HISTORY_EVENTS_PROMPT.md` 和 `HISTORICAL_EVENTS_IMPLEMENTATION_PLAN.md` 为主任务；下面内容是开发完成后的追加验证任务，不替代原要求。

## 执行时机与并发纪律

1. 不要在源码仍被其他代理修改时运行最终统计。先完成当前批次、通过定向测试，再冻结一个明确的代码快照。
2. 同一时间只能有一个代理写源码。并行代理默认只读代码，只能写各自独立的报告或临时输出目录。
3. 每个并行任务使用独立目录，例如 `artifacts/hy-history-events/<worker-id>/`。不得共用会被覆盖的 JSON、截图、Chrome 用户目录或日志文件。
4. 纯引擎、迁移、经济和战斗模拟可以高并发；系统 Chrome 只能串行或最多两路并发。
5. 不使用 Codex 内置浏览器、内嵌浏览器标签页或其桥接。真实 UI 只使用仓库现有系统 Chrome 流程。
6. 大规模运行前先执行小型 smoke matrix。若 smoke 失败，先修复，不要继续烧数万条无效样本。
7. 每个失败必须保留：代码快照标识、事件、立场、阵容、种子、波次、操作序列、期望、实际结果和最小复现。

## 建议并行工作组

可以并行安排六个只读验证工作组：

1. 确定性、存档、检查点与模糊操作。
2. 经济事件与经济曲线。
3. 五月风暴 9/10 人口和阵容搜索。
4. 七个事件与合法立场的战斗矩阵。
5. 世界大战与战争机器压力测试。
6. UI 合同、可访问性和系统 Chrome 串行验收。

各组只能提交报告和最小复现；由一个协调者统一修复源码、重跑定向测试，再重新冻结快照。

## 第一组：确定性、迁移和模糊操作

### 确定性双跑

- 所有事件和合法立场组合至少运行 100 个固定种子。
- 每个样本从相同输入连续执行两遍，对规范化完整结果做字节比较和 SHA-256。
- 比较事件 ID、候选及顺序、随机游标、奖励账本、逐波标志、棋子、金币、等级、经验、核心生命、战斗汇总和最终存档。
- 再执行“保存→加载→继续”和“失败→重试→继续”，必须与不打断的基准结果一致。
- 不得把 `Date.now()` 生成的棋子 ID直接纳入确定性哈希；先规范化不影响规则的运行时 ID。

### 随机动作模型测试

至少生成 10,000 条操作序列，每条 50—300 步，随机混合：

- 购买、刷新、购买经验；
- 部署、交换、撤回、王座；
- 出售；
- 事件确认、宗教改革领取；
- 改良主义替换；
- 立场选择；
- 自由主义完整退款；
- 开波、推进、暂停无关状态；
- 自动保存、手动保存、加载；
- 失败重试、本波重开；
- 重复点击和非法操作。

持续断言：

- 金币、经验、人口和次数均有界；
- 棋子 ID 与格子占用唯一；
- 事件、候选和立场不会重抽；
- 一次性奖励不会重复；
- 免费刷新、替换和完整退款不会跨波累计；
- 免费棋子参与合成后不会凭空增加 `paidCost`；
- 战斗临时字段不会进入稳定存档；
- 非法操作返回失败且不改变状态；
- 输入状态不被原地修改。

### 存档迁移语料库

生成至少 5,000 份存档：

- 无版本、V1—V7、未来版本；
- W1—W10、W3/W6 前后；
- 准备、战斗中、失败、胜利、完整通关；
- 缺字段、重复 ID、非法位置、非法事件/立场、候选重复；
- NaN、Infinity、负数、超大值、错误类型和截断数组；
- 缺少事件种子、游标、奖励账本、逐波标志；
- V6 不同星级棋子的购入成本迁移。

要求：合法旧档安全迁移；已越过触发点不补弹；未来版本明确拒绝；损坏数据被清洗且不崩溃；相同旧档总是派生相同事件种子。

### 长循环污染测试

- 连续执行至少 500 次“开波→推进→失败→重试”。
- 连续执行至少 200 次“结算→本波重开→再次结算”。
- 检查随机游标、奖励、经济账本、事件队列、状态、敌人、结构、延迟装置和检查点无累积漂移。
- 记录进程内存，判断是否存在明显线性增长；不要仅凭一次 GC 前后的数值下结论。

## 第二组：经济事件与经济曲线

矩阵至少覆盖：

- 策略：存钱、早期刷新、快速升级、随用随花；
- 阵容：平衡、法国混搭、英国证据，以及低费追三路线；
- 规则：普通、工业革命普通/激进、资本积累普通/激进、共产主义、自由主义；
- 每格 100—500 个固定种子。

输出：

- W2、W3、W5、W6、W8、W10 的平均/中位/分位金币；
- 满息首次波次、满息比例、金币上限溢出；
- 购买、刷新、升级花费；
- 工业革命消费门槛触发率、免费刷新利用率和作废率；
- 资本积累相对普通局何时落后、追平、开始净收益、十波回本率；
- 共产主义公共供给的相对收益；
- 自由主义退款金额、使用率及是否出现套利；
- 等级、人口、二星/三星和核心阵容成型时间；
- 通关率、核心生命和阵亡。

必须提供暂定数值调整前后对照，不得只交最终数字。发现数值异常时只调整集中配置允许的参数，不添加额外开局金币或新机制。

## 第三组：五月风暴与阵容搜索

为普通 9 人口和激进 10 人口建立固定阵容与搜索测试：

- 四大阵营纯阵容；
- 大阵营次高档混搭；
- 多小羁绊混搭；
- 德国黑格尔体系；
- 法国三个既有节点；
- 英国研究；
- 古希腊讲席；
- 普通前排、高阻挡、快速清兵、治疗和护盾结构。

至少：

- 每个手工阵容运行 100 个种子；
- 自动生成数千个合法 9/10 人口候选；
- 对候选进行 W5、W8、W9、W10 评估；
- 对表现最好的前 20 个阵容用更多种子复核；
- 检查是否存在跨事件、波次和种子都稳定占优的唯一十人口答案。

断言四大阵营有效人数封顶只影响大阵营档位；超过四人的成员仍战斗、施法并参加小羁绊。保守主义取消额外人口时不得删除或出售棋子，超人口必须阻止开波并报告需撤回人数。

## 第四组：七事件与立场战斗矩阵

不要依赖自然随机抽中事件。测试夹具应强制指定事件与合法立场，但不得暴露为玩家功能。

建议深度矩阵：

- 7 个事件 × 所有合法立场；
- 12 个代表阵容；
- 关键波次 W3—W10；
- 每个组合 100 个种子；
- 普通与激进版本分别统计。

特别检查：

- 城邦危机只冻结每路一名基础阻挡为 1 的地面棋子；高阻挡不增强；
- 宗教改革仅当前可用、不同大阵营、2 费；满备战区不丢奖励；
- 法国大革命只按费用改攻击和最大生命，不改技能、羁绊、费用或售价；
- 工业革命出售不倒扣消费，免费刷新不能自我生成；
- 保守主义同时停止持续收益与代价；
- 激进主义使用事件专属配置，不使用统一倍率；
- 改良主义替换结果不能再次替换；
- 资本积累不能生成共产主义；宗教改革不能生成保守或激进。

输出每个事件/立场的胜率、平均核心生命、核心生命分布、阵亡、战斗时长、经济和异常阵容。指出是否存在固定最优事件或立场，但不要只根据样本平均值直接改设计。

## 第五组：世界大战与战争机器

分别对 W4、W7、W9 运行普通、保守、激进版本，每种至少 500 个样本。阵容覆盖：

- 普通单前排；
- 多普通前排；
- 高阻挡；
- 快速清兵；
- 治疗续航；
- 护盾；
- 控制；
- 三路分守；
- 集中 A/B 汇合；
- 脆弱高输出。

记录：

- 战争机器数量和入口路线；
- 移动减伤持续时间；
- 首次被阻挡时间；
- 每台召唤次数、召唤时间和增援路线；
- 被哪些普通/高阻挡棋子接住；
- 核心伤害、阵亡和清场时间；
- W4/W7/W9 奖励次数和金额；
- 重试、读档和重复结算后的奖励状态。

强制断言：

- 普通前排可以接住一台超重战争机器；
- 移动减伤在被地面棋子阻挡后解除；
- 召唤次数严格封顶；
- 增援继承正确 `sourceRouteId`；
- W9 两台只发一份奖励；
- 保守主义取消 W7/W9 但不追回 W4；
- W5 洞穴之影、W10 绝对精神和世界之夜的既有行为与基线完全一致。

## 第六组：UI、可访问性和系统 Chrome

代码与完整自动门禁通过后再执行。系统 Chrome 每轮使用唯一端口和独立用户目录，建议 12—20 轮，不要启动 100 个浏览器。

流程覆盖：

- 新局到 W3，事件弹层出现且不能跳过；
- 刷新页面、自动存档和手动存档后仍是同一事件；
- 宗教改革备战区满时安全待领取；
- W6 三个候选顺序稳定，选择后不能重选；
- 当前事件/立场在紧凑状态区域持续可见；
- 工业革命进度与免费刷新；
- 改良主义商店替换；
- 自由主义本波使用状态；
- 9/10 人口部署、保守主义取消后的撤回提示；
- W4/W7/W9 战争机器预告、Boss 名称和奖励；
- 购买、刷新、替换、出售、拖放、开波、重试和存档加载；
- 1920×1080 与 1366×768；
- Console 无新增错误或警告。

补充可访问性检查：

- 弹层具备正确 dialog 语义、可见标题和按钮名称；
- 键盘 Tab 顺序可完成事件确认和立场选择；
- 焦点不会落到弹层背后的地图/商店；
- 关闭或选择后焦点回到合理入口；
- 状态不只依赖颜色表达；
- 中文与英文稳定 ID 不混入玩家文案。

若系统 Chrome 不可用，明确写“真实浏览器验收未验证”，不要以 SSR、源码检查或渲染字符串代替真实 DOM 验收。

## 覆盖率与变异测试

在不污染主工作区的临时副本中做测试有效性检查：

- 临时破坏事件抽取游标推进；
- 临时删除奖励幂等判断；
- 临时使 W9 双 Boss 各自发奖；
- 临时让 V6 W7 存档补弹；
- 临时让免费棋子按卡费累计 `paidCost`；
- 临时让大阵营封顶同时影响小羁绊；
- 临时去掉战争机器召唤上限。

对应测试必须失败。若变异仍通过，说明测试只有路径覆盖、没有行为约束，需要补强测试。变异只存在于可丢弃临时副本，不得提交或复制回主工作区。

## 静态审计脏活

完成后全仓搜索并报告：

- 历史事件抽取路径是否直接调用 `Math.random`；
- 事件数值是否散落在 UI、战斗循环或多个文件；
- React 本地状态是否成为规则来源；
- 奖励是否存在绕过 `claimHistoricalReward` 的路径；
- `paidCost` 是否在免费奖励、合成、存档和出售路径中一致；
- 是否误改 W5/W10、地图坐标、20 个部署 ID、商店数量或角色技能；
- 是否出现未使用的事件字段、重复配置或不可达分支；
- 玩家可见名称是否保持“往哲荣耀 / Philosophy Auto Chess”，兼容键是否仍保留旧 `idea-garrison-*` 标识。

## 分层运行规模

为避免一开始就产生巨量无效输出，按三层执行：

### L1 Smoke

- 每个事件和合法立场 3—5 个种子；
- 100 条随机操作序列；
- 50 份迁移语料；
- 每个战争机器波次 10 个样本。

### L2 Deep

- 每个核心组合 100 个种子；
- 10,000 条随机操作序列；
- 5,000 份迁移语料；
- 世界大战每种 500 个样本；
- 经济每格 100—500 个种子。

### L3 Soak

只在 L1/L2 全绿后运行：

- 500 次失败重试循环；
- 200 次胜利结算后本波重开；
- 重点经济组合 1,000 个种子；
- 前 20 个 9/10 人口候选扩大复核；
- 系统 Chrome 12—20 轮串行回归。

## 失败处理

发现失败时：

1. 保存原始失败样本。
2. 自动二分或缩减操作序列、阵容、波次和种子。
3. 判断是实现错误、测试错误、暂定数值风险还是环境噪声。
4. 只由协调者修改源码。
5. 先重跑最小复现，再跑该组 L1，最后恢复 L2。
6. 不得通过删除断言、扩大容差或跳过种子让测试变绿。

## 输出要求

原始结果写入机器可读 JSON/CSV，不在聊天中倾倒巨量日志。最终只汇总：

- 代码快照和运行命令；
- 各矩阵样本数；
- 确定性 SHA-256；
- 通过/失败/环境中断数量；
- 失败种子与最小复现路径；
- 经济曲线和关键分位数；
- 胜率、核心生命、阵亡和清场时间；
- 工业革命触发率、资本积累回本点；
- 战争机器召唤与奖励统计；
- 是否出现固定最优事件、立场或十人口阵容；
- 系统 Chrome 真实验收结果；
- 仍需真人确认的数值；
- 明确说明未打包、未发布、未上传 GitHub。

所有结论都要区分：已实现并验证、已实现但数值待调、仅测试夹具、尚未实现、环境未验证、已延期或否决。
