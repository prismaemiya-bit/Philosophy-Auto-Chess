### 现有 P0/P1 手机问题

#### P0：方向策略与本轮要求冲突

- `LandscapeGuard` 当前从根布局全局渲染，在 600px 以下竖屏用全屏层遮断全部页面。
- 这会让竖屏主菜单、设置、版本说明不可用，并尝试 fullscreen 与 `screen.orientation.lock(...)`。
- 必须改为：只在已进入对局的手机竖屏显示可关闭建议；不锁方向、不进全屏；关闭后用 `sessionStorage` 保证本次会话不重复。

#### P0：横屏仍不是单视口战场中心

- 844×390 实测截图中，顶栏占据约 70px，状态/共鸣栏继续占高，地图只显示上半部；底部经营区位于首屏以下。
- 667×375 会比 844×390 更紧，当前纵向文档流会同时出现页面滚动、地图和经营区局部滚动，战斗中难以观察全场。
- 932×430 空间略宽但仍会把桌面面板顺序纵向堆叠，没有形成真正的底部抽屉。
- 必须在手机横屏建立 `100dvh` 单屏网格：精简顶栏、地图占剩余中心区域、经营区以固定底部抽屉覆盖/收起；次级状态和面板按需打开。

#### P1：普通出售没有触控替代

- 普通出售仅在拖动时显示“拖入出售”；自由主义全退按钮不等于普通出售。
- 手机滚动和拖动容易冲突，不能把 HTML5 拖放作为唯一入口。
- 应在选中棋子时提供明确的手机操作条，包含取消选择和出售；出售需二次点按确认，避免滑动/误触直接卖出。

#### P1：撤回动作不够可发现

- 从战场点棋子后再点空备战位，底层逻辑可以撤回，但当前没有清楚提示目标和动作。
- 选中状态需要更明显；手机操作条应说明“再点地图/备战格移动或交换”，并提供显式撤回到首个空备战位的快捷按钮。

#### P1：顶栏触控目标过多

- 手机横屏仍同时显示金币、人口、波次、核心、开波/暂停、任务、设置，且每项固定宽度后横向滚动。
- 横向滚动会隐藏开波或设置，不符合核心操作随时可见的要求。
- 横屏顶栏应保留核心生命、金币、人口、波次和开波/暂停；任务、设置、历史说明进入紧凑“更多”区域或保持较小但可点的次级按钮。

#### P1：弹窗和结算缺少低高度专项约束

- 历史事件弹窗虽使用 `100dvh`，但选择卡、说明和按钮在 375px 高度下需要确认内部滚动和底部按钮可达。
- `decision-overlay`、共鸣详情、教程、设置、任务、单位详情、胜败结算分别由不同 CSS 文件控制，存在层级与滚动所有者不一致的风险。
- 胜利页在窄屏已有纵向滚动，但 667×375/844×390 必须验证重新开始、查看战绩、四榜切换和关闭按钮均在滚动容器内可达。

#### P1：点击与拖放的状态边界

- 屏幕旋转不会重建 React 状态，所以选择理论上会保留；但抽屉/弹窗几何会立即变化，当前无 resize 后可见性恢复断言。
- HTML `draggable` 在触屏浏览器实现不一致；必须保留桌面拖放，同时把触屏主路径确定为点选。
- 快速购买当前使用渲染闭包中的 `state`，连续极快点按有陈旧状态风险；刷新、冻结等已部分使用 `stateRef`。手机应给购买/出售/历史确认增加 UI 侧短暂互斥，核心规则仍由引擎拒绝重复。
- 页面拖动时，普通点按事件可能在手指抬起时触发；主要滚动区需要 `touch-action: pan-x` 或 `pan-y`，战场点击区使用 `manipulation`，不能全局禁止必要滚动。

### 分尺寸审查

#### 360×800、390×844 竖屏

- 当前：全局强制横屏层完全遮挡菜单与设置；390×844 系统 Chrome 截图已确认。
- 修改目标：主菜单、任务、版本说明和设置使用单列滚动；进入对局只显示底部横屏建议，可继续操作或关闭。
- 战场保留 680px 逻辑宽度并在地图容器内横向浏览，不能让整个页面产生 `scrollX`。
- 商店和备战区需要局部横向滚动；弹窗高度使用 `100dvh` 与安全区。

#### 667×375 横屏

- 当前风险最高：顶栏、状态栏、共鸣栏和地图按文档流叠加，地图及经营区不能同时进入视口。
- 地址栏展开后可用高度可能低于 375px；`100vh` 会产生底部跳动，应由手机层统一使用 `100dvh`。
- 目标是地图最少获得约 250px 可视高度；经营抽屉默认收起或仅显示标签，展开后覆盖地图下部并可内部横向滚动。

#### 844×390 横屏

- 当前系统 Chrome 能完成购买、点选部署、开波与暂停，但截图只显示地图上半部，属于“功能可达、布局未完成”。
- 目标是首屏完整看到 16:9 战场；经营抽屉展开时允许有意覆盖部分地图，但收起后地图必须完整。

#### 932×430 横屏

- 空间足够保留完整精简顶栏、状态短句和约 330px 高战场。
- 商店五张卡不应被压成极小字号；应继续用横向卡片轨道，经营按钮固定在同一抽屉内。

#### 常见安卓浏览器缩放与动态地址栏

- 根页面禁止意外横向滚动；局部地图/商店/备战滚动拥有明确滚动容器。
- 不使用固定像素 `100vh` 作为手机游戏高度；以 `100dvh` 为主，`100vh` 只作旧浏览器回退。
- safe area 需覆盖顶栏、方向建议、底部抽屉和所有全屏弹层。
- 不全局禁用缩放；保留初始缩放 1，不设置 `maximum-scale` 或 `user-scalable=no`。用 `touch-action: manipulation` 防无意义双击缩放，用局部 `pan-x`/`pan-y` 保留阅读滚动。

### Hover、键盘、浏览器手势和生命周期

- 商店卡、槽位、按钮有大量 `:hover` 装饰，但核心信息与功能不依赖 hover；手机需用 `.selected`、`:focus-visible` 和 `aria-pressed` 提供等价反馈。
- 游戏没有文本输入主流程，横屏键盘影响主要限于开发工具；生产手机布局应保持开发工具折叠。
- 浏览器返回手势目前会正常离开页面，没有对局内确认层。直接拦截 History API 风险较高，本轮先保证 `pagehide` 稳定保存并记录真机验证，不擅自改变浏览器返回语义。
- 页面切后台后战斗定时器可能被浏览器节流；稳定存档不会保存临时战斗。是否自动暂停属于玩法体验决策，本专项不修改战斗规则，先列为真机验收项。

### PWA 兼容性初判

- 当前 vinext/Cloudflare 构建能直接发布 `public/` 静态资源，Web App Manifest 与静态 Service Worker 注册路径可行。
- 不应缓存应用 HTML、RSC 响应或构建产物，否则可能长期停留在旧存档迁移代码。
- 本轮可安全完成 manifest、standalone、主题色、iOS metadata、正式图标复用，以及“只缓存独立离线说明页”的保守 Service Worker。完整离线游玩留到后续专门版本化资源清单后实现。

### 阶段一结论

- 桌面版自动化基线稳定；手机初稿尚未满足本轮目标。
- 本轮采用同一 React/引擎逻辑、独立手机 CSS 与少量移动交互接线；不复制规则、不改平衡。
- 下一步优先级：移除全局方向门禁 → 建立横屏单视口与底部抽屉 → 增加选择操作条/安全出售 → 统一弹窗低高度滚动 → 添加 PWA 基础 → 扩展系统 Chrome 手机断言。
